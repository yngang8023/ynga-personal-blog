第一优先级：命中置信度判断
现在你已经有“检索不到就回退普通回答”，但还不够。下一步应该看“检索到了，但命中得不够好”。
建议加：

rerank score threshold
top1-top2 score gap
有效 source 数量
是否全是同一篇文章重复 chunk
这一步价值最高。它能明显减少“看起来答了，其实没答准”。

第二优先级：把 metadata 真正用起来
你现在有 title / tags / category / updated / images 这些素材，但检索时还没充分利用。
对博客很适合加：

hasImages
hasCodeBlocks
category
tags
updatedAt
series / topic
然后做两类动作：

过滤：比如问“截图”“图示”“流程”，优先只查 hasImages=true
提升：比如问“最新”“现在”，优先 boost updatedAt desc
这和 Cloudflare 官方的 metadata filtering / boosting 思路是一致的。

第三优先级：对话历史参与 query rewrite
你现在的 rewrite 主要还是基于最后一句问题。
但实际用户会连续问：

“那怎么部署？”
“这个报错怎么处理？”
“有截图吗？”
这种 follow-up 问题，如果不带上历史语义，rewrite 很容易弱。Cloudflare 官方也明确说了 query rewriting 对 follow-up query 特别有用。

第四优先级：Parent-Child 检索
这个很适合你的教程博客。

做法：

小 chunk 用来召回
大父块用来生成
比如：

子块：300-400 tokens，命中具体问题
父块：同一章节 1000-1500 tokens，保留上下文、步骤、注意事项
这样比单纯把 Top-K 拉大更稳，也更适合“部署流程 / 配置教程 / 报错排查”这类长文。

第五优先级：图片检索从“挂在 chunk 上”升级到“可判定相关性”
你现在图片已经能展示，但更聪明的方向是：

给图片单独存 caption / OCR / surrounding text
检索时判断“这张图是否真的回答了这个问题”
一个回答里限制最多 1-2 张强相关图片
不然容易出现“有图就插”，但图并不帮助回答。

第六优先级：上下文压缩
现在召回后，还是把 chunk 比较直接地送给生成模型。
可以继续做：

每个 source 先压成 短证据摘要
去重相似 chunk
同一文章最多保留 2 个关键 section
优先保留“步骤 / 配置 / 结论 / 注意事项”段落
这会比继续堆 chunk 数量更有效。
